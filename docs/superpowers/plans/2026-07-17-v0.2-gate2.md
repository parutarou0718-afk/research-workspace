# v0.2 Gate 2 Incremental Monitoring and Version Candidates Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver Gate 2 only: commercially cleared incremental monitoring, persistent event/debounce/reconciliation state, deterministic bounded PaperVersionCandidate generation, and read-only monitoring/candidate UI.

**Architecture:** Filesystem events only wake persistent checks; they never mutate formal observations. Feature-specific workers perform bounded observation, stability, reconciliation, and candidate computations, return immutable DTOs, and the existing Application/write-coordinator boundary validates and persists facts plus DomainEvent 2.0. Candidate generation uses closed deterministic rules and indexed local neighborhoods, while all formal version decisions remain deferred to Gate 3.

**Tech Stack:** Python 3.12, PySide6, SQLAlchemy 2.0, Alembic, SQLite, watchdog, jsonschema Draft 2020-12, RFC 8785 canonical JSON, SHA-256, pytest/pytest-qt/pytest-socket.

**Normative source:** `docs/superpowers/specs/2026-07-16-v0.2-deterministic-design.md` at commit `f55fb98ed6b8723507fc4564ba614dca0aa249a8`.

**Required baseline:** Gate 1 checkpoint `25c5230000511c5982a669c18fe4b636edeecb36`. Packaging work is unrelated and remains paused; Gate 2 must not create or depend on a frozen application bundle.

## Global Constraints

- Implement Gate 2 only. Gate 3/4 tables, commands, controls, placeholder columns, and empty skeletons remain absent.
- Do not implement candidate confirmation/rejection/reconsideration, formal version edges, Paper/Idea/Submission CRUD, undo, recovery points, backup/restore/export, AI, OCR, Agent, TaskExecutor, plugins, or network access.
- Original sources and committed snapshots remain read-only. Monitoring has read/list/copy authority only and reuses the Gate 1 immutable snapshot workflow.
- Use `RawFileEvent → PendingPathCheck → SourceObservation`; callbacks cannot directly change formal source state.
- Healthy operation performs no periodic full traversal. Reconciliation occurs only for approved bounded reasons.
- Adding a root creates metadata-only baseline facts. Baseline files are not hashed, snapshotted, parsed, or considered for candidates.
- Same content at different paths preserves every provenance observation and never creates a new snapshot or version candidate.
- Version candidates are immutable suggestions. Gate 2 UI is read-only for candidate decisions.
- All blocking filesystem, hash, reconciliation, and detector work runs outside the Qt thread. Workers own no Session, repository, QWidget, Qt model, unrestricted path, or network capability.
- Every new long-lived page uses a separate `.ui`; Python must not construct layouts.
- New DomainEvents use closed 2.0 payloads and commit atomically with their database facts.
- Tests must first fail for a genuine unmet requirement. Do not manufacture RED, weaken tests, skip, or xfail.
- After every GREEN task run the focused command and `uv run pytest -q`.
- Every Task ends in its own reviewable commit and task evidence. Human approval is grouped at the Checkpoints below; execution may continue only inside an explicitly approved Checkpoint group.
- `REL-GATE-001` remains `BLOCKED_BY_ENVIRONMENT` and blocks only public packaged release.

## Task, Commit, and Checkpoint Policy

- A **Task** is the smallest implementation/RED-GREEN unit and always produces one independently revertible commit.
- A **Checkpoint** is the human approval boundary covering one or more completed Task commits.
- A failure in any Task prevents later Tasks in the same Checkpoint from starting.
- At a Checkpoint, report every Task commit separately plus the accumulated suite. Do not squash Task commits.
- A later Checkpoint cannot begin until the current Checkpoint evidence is approved.

## Fixed Interface Ledger

The detailed implementation may use deeply immutable tuples or canonical JSON bytes, but it must preserve these names and semantics:

```python
@dataclass(frozen=True)
class RawFileEventDTO:
    event_id: UUID
    monitoring_root_id: UUID
    provider: str
    event_type: Literal["created", "modified", "moved", "deleted", "overflow", "root_state"]
    source_path: Path | None
    destination_path: Path | None
    observed_at: datetime
    ingested_at: datetime
    correlation_hint: str | None
    deduplication_key: str

@dataclass(frozen=True)
class PendingPathCheckDTO:
    pending_check_id: UUID
    monitoring_root_id: UUID
    normalized_path: Path
    first_event_at: datetime
    last_event_at: datetime
    merged_event_types: tuple[str, ...]
    state: str
    stability_attempt_count: int
    next_check_at: datetime | None
    row_version: int

@dataclass(frozen=True)
class ReconciliationPlan:
    operation_id: UUID
    reconciliation_run_id: UUID
    monitoring_root_id: UUID
    reason: Literal["baseline", "disconnect", "overflow", "unclean_shutdown", "user_verify"]
    root_path: Path
    checkpoint: bytes | None
    page_size: int

@dataclass(frozen=True)
class CandidateInput:
    snapshot_id: UUID
    sha256: str
    mime_type: str
    first_seen_at: datetime
    observation_ids: tuple[UUID, ...]
    filename: str
    normalized_parent_path_hash: str | None
    title: str | None
    normalized_text: str | None
    zero_text: bool

@dataclass(frozen=True)
class CandidateDetectionResult:
    earlier_snapshot_id: UUID
    later_snapshot_id: UUID
    detector_id: str
    detector_version: Literal["1.0"]
    rule_id: Literal[
        "R1_SOURCE_CONTINUITY",
        "R2_REPLACE_CONTINUITY",
        "R3_PAPER_TITLE_TIME",
        "R4_NAME_TITLE_TEXT",
        "R5_ZERO_TEXT_LINEAGE",
    ]
    rule_config_fingerprint: str
    direction_rationale: bytes
    signals: bytes
    input_observation_ids: tuple[UUID, ...]
```

`bytes` above means canonical immutable RFC 8785 JSON, not arbitrary JSON or mutable mappings.

## Checkpoint Evidence Format

Each Task records the following evidence when its commit is created. The
Checkpoint report presents every included Task separately and then the
accumulated result:

1. Commit hash and branch.
2. Exact files changed.
3. Literal focused RED command/result and genuine failure reason.
4. Literal focused GREEN command/result.
5. Literal accumulated `uv run pytest -q` summary.
6. Acceptance criteria newly satisfied.
7. Database/event/file/thread/privacy evidence relevant to the task.
8. Known blockers or deviations.
9. `git diff --check` and literal `git status --short`.
10. Confirmation that later-task files and behavior remain absent.

The final Gate 2 report additionally includes:

- G2-AC01–G2-AC21 status table with exact test evidence.
- Alembic head and schema inventory.
- Watchdog exact lock/transitive/license/notices evidence.
- Raw-event replay, disconnect, restart, reconciliation, and no-periodic-scan evidence.
- Candidate rule/revision/scale evidence.
- UI ownership, threading, DPI, unread, and read-only-control evidence.
- Original supplied-file SHA-256 comparison.
- Confirmation that no test was weakened, skipped, or marked xfail to obtain green.
- `REL-GATE-001 — BLOCKED_BY_ENVIRONMENT` carried unchanged.
