This document is the single normative specification for v0.2. It replaces no
v0.1 rule unless the replacement is stated explicitly. Implementation is
forbidden until this complete document and the Gate 1 detailed RED/GREEN plan
are separately approved.

## 1. Product objective and release gates

v0.2 adds deterministic local research-data workflows while preserving the
v0.1 privacy, evidence, audit, Qt ownership, and offline-first boundaries.
Work is divided into four independently reviewable stage gates:

1. **Gate 1 — Immutable import and deterministic parsing:** read-only import,
   immutable content-addressed snapshots, DOCX/PDF/PPTX adapters, fingerprints,
   duplicate detection, ParseArtifact revisions, and read-only import UI.
2. **Gate 2 — Observation and version candidates:** incremental monitoring,
   persistent event recovery, bounded reconciliation, deterministic explainable
   PaperVersionCandidate generation, and read-only candidate UI.
3. **Gate 3 — Protected domain editing:** Paper/Idea/Submission CRUD, relation
   decisions, formal PaperVersion DAG, command audit, compensating undo, legacy
   migration, and two-slot verified pre-command recovery protection.
4. **Gate 4 — Portability and release acceptance:** research-data export,
   complete backup/restore, restored-workspace activation, accumulated
   regression, commercial-license evidence, and final v0.2 acceptance.

Each gate requires its own reviewed detailed implementation plan before its
production work begins. Each gate ends with a checkpoint commit and an evidence
report containing exact changed files, tests added, tests run and literal
results, satisfied acceptance criteria, and blockers/deviations. No later gate
begins before the preceding checkpoint is approved.

REL-GATE-001 remains `BLOCKED_BY_ENVIRONMENT`. It blocks a public packaged
release only and does not block v0.2 internal development. v0.2 cannot be called
public-release-ready until mixed-DPI Windows verification is completed.

## 2. Global scope and non-goals

### 2.1 Included

- Read/list/copy-only access to explicitly selected external research files.
- Immutable, content-addressed local copies using SHA-256 physical identity.
- Deterministic DOCX, PDF, and PPTX parsing into ParsedDocument 2.0.
- Provenance-preserving physical deduplication.
- Explainable deterministic version candidates and confirmed version DAGs.
- Incremental filesystem sensing with persistent recovery and bounded repair.
- Paper, Idea, and Submission CRUD only.
- Explicit relation confirmation, rejection, reconsideration, and retraction.
- Field-level optimistic concurrency, command idempotency, audit, and undo.
- Verified write-before recovery points for approved user domain commands.
- JSONL/CSV research-data export and independent full backup/restore.

### 2.2 Excluded

- OCR, AI, embeddings, semantic search, context recovery execution, automatic
  Idea extraction, arbitrary Task execution, Agent execution, plugin execution,
  network providers, cloud conversion, or external services.
- CRUD expansion for Note, Conference, or Grant.
- Tags, arbitrary custom fields, generic metadata bags on user CRUD entities,
  rich-text storage, citation-manager features, or SubmissionRound.
- Automatic confirmation of Paper versions or relations.
- Source-file writes, moves, renames, deletions, or archival.
- Automatic source-snapshot or parse-attempt garbage collection.
- Periodic full traversal during healthy monitoring.
- Permanent deletion, force undo, redo, undo-of-undo, database time travel, or
  workspace merge.
- `.rwsexport` re-import, ZIP64 backup transport, backup encryption, incremental
  backup, and in-place restore.
- Unfinished, disabled, decorative, or placeholder interactive controls.

## 3. Normative principles

1. Import success means an immutable SourceSnapshot was safely committed.
   Parsing success is independent.
2. SHA-256 identifies physical bytes. Names and paths are provenance only.
3. One physical snapshot exists per hash; distinct source observations remain
   distinct.
4. SourceSnapshot, ParseArtifact, SourceDocument, ParsedBlock, and EvidenceRef
   are separate layers. Old evidence never follows a new parser revision.
5. Candidates are explainable suggestions. Only protected user commands create
   formal domain facts.
6. Recovery points protect pre-command state; compensating commands implement
   ordinary undo. Neither substitutes for the other.
7. Backup restores a workspace. Export exposes research data. Their package
   types and acceptance paths are disjoint.
8. Background execution is deterministic worker execution, not Agent execution.
9. Original sources and committed snapshots are immutable to application
   workflows.
10. Diagnostic logging never contains research text, evidence excerpts,
    credentials, PDF passwords, or portable full external paths.

## 5. Locked data-directory layout

```text
<data-directory>/
├─ research_workspace.db
├─ sources/
│  └─ sha256/<first-two-hex>/<sha256>/content
├─ derived/
│  └─ parse/<parse-artifact-id>/parsed_document.json
├─ staging/
│  ├─ imports/
│  ├─ parse/
│  ├─ backup/
│  ├─ export/
│  └─ restore/
├─ recovery/
│  ├─ current/{workspace.db,manifest.json}
│  └─ previous/{workspace.db,manifest.json}
├─ exports/
├─ backups/
└─ logs/
```

Committed source files are extensionless because the hash, MIME type, and
database record carry identity. The path is deterministic but is not part of
content identity. `sources`, `derived`, `staging`, `recovery`, `exports`, and
`backups` are never valid import or monitoring roots. `recovery` is internal,
not displayed as a user backup, and is not included as such in Gate 4 packages.

Data-directory switching keeps the v0.1 restart model. Selecting a different
directory writes `pending_data_directory`; it does not migrate, merge, copy, or
delete existing data. Bootstrap validates the pending directory and falls back
to the prior active directory on failure.

## 6. Common persistence rules

- Table and column names are lower snake case; tables are plural.
- IDs are lowercase canonical UUID strings unless explicitly declared SHA-256.
- Timestamps are UTC RFC 3339 text ending in `Z`.
- JSON is UTF-8 RFC 8785 canonical JSON text; Python serialization is forbidden.
- Mutable domain rows use integer `row_version >= 1` and atomic compare/update.
- Research data is never cascade-deleted. Foreign keys use `RESTRICT`, except a
  specifically approved optional pointer may use `SET NULL`.
- SQLite foreign keys are enabled on every connection; WAL mode is supported.
- Cross-polymorphic and graph invariants are checked before work and again in
  the protected transaction. Complex SQLite triggers are not used as a
  substitute for application invariants.
- A migration first creates a SQLite Online Backup pre-migration image, then
  validates counts, schema, constraints, and integrity before success.

## 7. Source retention and import

### 7.1 Physical identity and provenance

`SourceSnapshot` is immutable physical content. `SourceObservation` is mutable
location availability and provenance. Multiple observations may reference one
snapshot. Missing, moved, renamed, or unavailable external paths never
invalidate a snapshot or historical evidence.

Snapshot creation is:

```text
pre-stat
→ open source handle
→ stream to same-volume staging while hashing
→ flush and fsync staging
→ post-stat and file-identity comparison
→ verify size and SHA-256
→ atomic promotion to deterministic hash path
→ transactional registration
```

If the source changes, the operation commits neither the staged content nor a
SourceSnapshot record. A new event may retry after stability. Existing snapshots
are never overwritten. Changed bytes create a new snapshot.

Symlinks, junctions, reparse points, mount redirections, unsafe final paths,
partial writes, and hash mismatches are explicit safe failures. External access
is `read/list/copy` only.

### 7.2 Import state

The public terminal `ImportOutcome` is `imported|failed` and is independent of
ParseAttempt status. Persistent scheduling uses a separate `ImportItemState`:
`pending|imported|duplicate_content|failed|cancelled`. `imported` and
`duplicate_content` map to public outcome `imported`; `failed` maps to `failed`;
pending/cancelled have no successful import outcome. These are not competing
definitions of one enum.
Import UI discloses that a local immutable copy is created and previews the
estimated added storage after known-hash deduplication where possible. Estimate
uncertainty must be displayed; preview does not hash a whole monitoring baseline.

### 7.3 Deduplication

- Same SHA-256 reuses the committed physical content.
- Every selected path still creates or updates its own observation.
- Same content at different paths is not a paper version.
- Deduplication never discards original filename, path history, availability,
  observation time, or monitoring-root provenance.
- Automatic snapshot garbage collection is out of scope.

## 8. Parse revisions and ParsedDocument 2.0

### 8.1 Identity graph

```text
SourceSnapshot
  └─ ParseArtifact
       ├─ ParseAttempt[]
       └─ SourceDocument
            └─ ParsedBlock[]
                 └─ EvidenceRef[]
```

One logical ParseArtifact is unique by snapshot, parser ID, parser version,
config fingerprint, and output contract version. It may have multiple immutable
attempts but at most one successful output. Failed/cancelled attempts have no
SourceDocument or ParsedBlock. A successful attempt requires contract-valid
output and atomic artifact/document/block registration.

The current preferred artifact is an audited pointer only. It never rewrites
PaperVersion, EvidenceRef, or earlier parse output. Parser/config/contract
changes create new artifacts.

### 8.2 Configuration fingerprint

Parser configuration follows a closed JSON Schema, expands all defaults,
normalizes strings, serializes with RFC 8785, and hashes with SHA-256. It includes
at least OCR enablement, headers/footers/footnotes policy, table encoding,
language, image-alt policy, reading-order strategy, adapter extensions, and
ParsedDocument contract version. Worker concurrency, UI refresh rate, log level,
and batch scheduling are operational settings and do not change artifact identity.

Gate 1 fixes `ocr_enabled=false`. Any request for OCR is rejected before parser
entry with `UNSUPPORTED_CONFIGURATION`.

### 8.3 ParsedDocument 2.0

The 2.0 contract replaces the v0.1 parser output identity semantics. Required
top-level keys are:

```text
schema_version = 2.0
parse_artifact_id
source {source_snapshot_id, sha256, mime_type, size_bytes,
        storage_relative_path}
parser {parser_id, parser_version, config_fingerprint, contract_version}
title
metadata
blocks[]
warnings[]
```

`storage_relative_path` is diagnostic/recovery information only. It is not in
ParseArtifact identity, block identity, or EvidenceRef identity. Restore may
regenerate it for the same snapshot.

Each block contains `block_id`, contiguous zero-based `block_index`, `kind`,
non-empty normalized `text`, closed `locator`, and open but contract-owned
metadata. `paragraph_index` is nullable and means an index in the applicable
native paragraph sequence, not the global block order. `char_start` and
`char_end` are a half-open range relative to that block; a full emitted block
uses zero and normalized-text length. Evidence may later select a subrange.

For `paragraph`, `list_item`, `caption`, `footnote`, `code`, `equation`, and
`image_alt`, `paragraph_id` equals the final deterministic block ID. For title,
heading, table, header, footer, and other blocks it is null. This rule is the v2
derivation; no adapter-native unstable paragraph identifier is substituted.

Block identity is SHA-256 of the unambiguous UTF-8 framed tuple:

```text
parse_artifact_id,
block_index,
kind,
RFC8785(locator without paragraph_id),
NFC(text with CRLF/CR normalized to LF)
```

Length-prefixed framing is used; delimiter concatenation is forbidden.

`output_sha256` hashes RFC 8785 canonical ParsedDocument after removing only
`source.storage_relative_path`, so internal layout does not change semantic
output identity. `derived_file_sha256` separately hashes the exact UTF-8
canonical `parsed_document.json` bytes including that diagnostic path. Restore
verifies the file hash and may regenerate the semantic-equivalent file only after
recomputing both values.

The common locator contains page, slide, block index, nullable paragraph index,
nullable paragraph ID, heading path, block-relative character range, nullable
source offsets, nullable bounding box, and `native_locator`. The latter is a
discriminated closed union with `type=docx|pdf|pptx`; validators never infer its
shape from MIME type.

DOCX native location records part, section/part order, body item, paragraph,
table/row/column/cell paragraph, style, and image relationship when applicable.
PDF records 1-based page and extraction index. PPTX records slide plus recursive
`shape_path[]`, each element holding shape ID and shape-tree index, along with
text paragraph, table cell, notes, and alt-source fields.

### 8.4 Adapter rules

**DOCX (`parser_id=python-docx`)** traverses body paragraphs and tables in OOXML
child order. Headers/footers follow section and part order when enabled. Library
limitations for footnotes, endnotes, text boxes, tracked changes, fields,
equations, SmartArt, and embedded objects produce stable warnings; the contract
does not claim unsupported coverage. No macro or external resource executes.

**PDF (`parser_id=pypdf`)** follows the approved PDF boundary:

- normal and empty-password PDFs may succeed;
- a non-empty password requirement retains the snapshot and fails the attempt
  with `PDF_PASSWORD_REQUIRED`; v0.2 never requests, accepts, caches, persists,
  or logs passwords;
- valid image-only PDFs succeed with SourceDocument, zero blocks, and warnings
  `NO_EXTRACTABLE_TEXT` and `OCR_REQUIRED`;
- corrupt, truncated, invalid-structure, unsupported-feature, and read failures
  map to stable product codes, not pypdf exception class names.

**PPTX (`parser_id=python-pptx`)** follows slide and OOXML shape-tree order,
recursing into group shape trees. Missing authoritative reading order produces
`PPTX_READING_ORDER_HEURISTIC`. Notes are configuration-controlled. Charts,
SmartArt, media, embedded objects, and unsupported constructs produce stable
warnings. Images emit only explicit title/description alt text and retain the
native shape locator.

DOCX/PPTX tables use `table_text_version=tsv-escaped-1`: columns use tab, rows
use LF, and cell backslash, tab, LF, and CR are encoded as `\\`, `\t`, `\n`,
and `\r` respectively, in that escape order. Tables emit once; overlapping cell
paragraph duplicates are forbidden.

### 8.5 Evidence stability

EvidenceRef stores parse artifact, parsed block, original locator, block-relative
character range, normalized quote hash, optional user-saved excerpt, and creation
time. Reparse never changes it. A future relocation workflow may suggest a new
block but must create a new EvidenceRef after explicit confirmation and retain
the old reference. Gate 4 retains every parse revision reachable from Evidence,
PaperVersion, AuditChange, BackgroundOperation result, or DomainEvent, plus the
current preferred artifact. v0.2 performs no automatic parse garbage collection.

Evidence quote normalization is exactly NFC after CRLF/CR→LF conversion, with no
trim or whitespace folding. `quote_hash` is SHA-256 of its UTF-8 bytes. A saved
excerpt, when present, must normalize to the same hash and match the referenced
block subrange at EvidenceRef creation; later source revisions do not revalidate
or rewrite it.

## 9. Incremental monitoring

### 9.1 Three-layer observation model

```text
RawFileEvent → PendingPathCheck → SourceObservation
```

Filesystem callbacks never mutate formal source state directly. Raw events are
append-only observations. Pending checks own debounce, stability, retry, and
reconciliation scheduling. SourceObservation changes only after verified facts.

Adding an explicit user-selected root creates a metadata-only observation
baseline: normalized path, type, size, modified time, optional file identity,
and baseline time. Baseline files are not hashed, snapshotted, parsed, or used in
version candidates. Historical import remains an explicit Gate 1 action; Gate 2
has no unfinished backfill control.

Nested or overlapping active roots are rejected after Windows final-path,
case-insensitive, and Unicode normalization. Internal workspace directories and
reparse/link targets are rejected. A moved event is only continuity evidence;
new-path stable reading and SHA-256 determine the resulting fact.

### 9.2 State machines

Root status:

```text
active | paused | disconnected | degraded | overflow_reconciling | error
```

`disconnected` means the root is unavailable. `degraded` means partial
permission/backend/overflow/reconciliation trouble while the root remains at
least partly accessible. `error` means the subsystem cannot safely continue.
Disconnection never marks all children missing.

Pending path status:

```text
detected → debouncing → waiting_for_stability → importing
         → imported | duplicate_content | safe_failure
waiting_for_stability → unstable_source
unstable_source → debouncing  (new event or explicit recheck)
```

The default semantic monitor configuration is: two stable stat observations, a
2-second quiet window, maximum five stability attempts, and backoff of 2, 5, 15,
30, and 60 seconds. These values, allowed extensions, exclusions, and candidate
window enter the configuration fingerprint. Worker count, UI refresh, and batch
size do not.

### 9.3 Recovery and reconciliation

Healthy operation performs no periodic full traversal. Root-scoped bounded
reconciliation occurs after baseline, queue overflow, disconnection recovery,
unclean shutdown, or explicit full material verification. It is paged,
cancellable, checkpointed, and first compares normalized path, size, modified
time, and persisted file hints. Only suspected new/changed files enter stable
hash import.

Clean shutdown stops intake, flushes RawFileEvent and PendingPathCheck state,
joins workers, and writes a clean marker. Startup checks watcher generation,
overflow marker, root availability, shutdown marker, and checkpoints.

RawFileEvent automatic destructive cleanup is out of v0.2. Required indexes are
root/time, normalized-path-hash/time, processing status/time, and correlation
hint. UI capacity warning begins at either 1,000,000 retained raw events or an
estimated 1 GiB raw-event table, whichever occurs first. This is a warning and
does not delete data.

### 9.4 Persistent attention and UI

Gate 2 shows roots, health, last event, waiting/failed/unread counts, recent
imports, reconciliation progress, and read-only version candidates. New facts
remain unread until the user actually opens their row/detail; refresh or
auto-selection cannot clear unread. Desktop popups are not a v0.2 dependency.

Implemented actions are add, pause, resume, remove root, recheck an exceptional
item, mark viewed, pause/resume reconciliation, and explicit material
verification. Removing a root stops future sensing only. Confirm/reject version,
source repair, history deletion, snapshot cleanup, and fake auto-fix controls are
absent.

## 10. Paper version candidates and formal graph

### 10.1 Candidate identity and direction

PaperVersionCandidate is an immutable explainable suggestion, not a formal
relation. Its unique identity is:

```text
earlier_snapshot_id,
later_snapshot_id,
detector_id,
detector_version,
rule_config_fingerprint
```

The rule fingerprint is SHA-256 over default-expanded RFC 8785 JSON. Candidates
require distinct snapshot hashes and compatible document types. Same-hash paths
are filtered before detection. Modified time alone never establishes direction.
Direction precedence is: consecutive stable observation, verified replace/move
continuity, first-observed order, same-Paper import order, filename version
sequence, then filesystem time as corroboration only. If deterministic direction
remains ambiguous, no directed candidate is created.

“Clear order” means the first applicable precedence source yields one strict
earlier/later result and no higher- or equal-precedence source contradicts it.
Equal timestamps, conflicting continuity, or incomparable filename tokens are
ambiguous. A consecutive observation means adjacent `verified|changed` facts for
one SourceObservation ordered by `(observed_at,id)`. Reliable replace/move means
one provider correlation inside one watcher generation whose source/destination
facts are both later stably verified; the event alone never proves bytes.

### 10.2 Versioned rules

The first detector registry is closed and uses stable rule IDs:

- `R1_SOURCE_CONTINUITY`: one SourceObservation changes to different content.
- `R2_REPLACE_CONTINUITY`: reliable replacement evidence plus changed hash.
- `R3_PAPER_TITLE_TIME`: same Paper, normalized title match, clear order.
- `R4_NAME_TITLE_TEXT`: normalized filename and title plus configured deterministic
  text-fingerprint threshold.
- `R5_ZERO_TEXT_LINEAGE`: same Paper plus path lineage, version suffix, and clear
  order for zero-text content.

Rule normalization is fixed for detector version 1.0. Titles use NFC, Unicode
outer trim, internal whitespace folding, and casefold; R3/R4 require exact
normalized-title equality. Filename lineage removes only the final extension,
applies the same normalization, folds space/underscore/hyphen/dot runs, and
separates—but does not discard—the final configured version token. The 1.0 token
registry is `draft`, `final`, `rev`, `revision`, `v`, `version`, `初稿`, `终稿`,
`最终`, and `修订稿`, optionally followed by an integer. Its exact list is part
of the rule config fingerprint.

Version-token direction is used only when both lineage keys match and tokens are
comparable. Stage order is `draft|初稿` < numeric revision families
`rev|revision|修订稿|v|version` < `final|终稿|最终`; two numeric tokens in the
same family order by integer. Numeric tokens in different families are
incomparable. A parent-path lineage match means equal normalized parent-path hash
inside the same MonitoringRoot; path text is not copied into the candidate.

Text fingerprint 1.0 uses the normalized extract (NFC, casefold, whitespace
folding), requires at least 200 code points, hashes every overlapping 13-codepoint
gram with SHA-256, retains the lowest 2,048 distinct 64-bit big-endian prefixes,
and computes Jaccard similarity. R4 requires equal filename-lineage key, equal
normalized title, and similarity >=0.80. R5 does not compute text similarity and
requires same Paper, a common SourceObservation or parent-path lineage, matching
filename-lineage key with distinct ordered version tokens, and deterministic
observation direction. All DOCX/PDF/PPTX snapshots are format-compatible; MIME
change is retained as an explanation signal rather than an exclusion.

Same-Paper membership alone is insufficient. Every candidate stores rule and
detector revision, normalized/original filename representations, title evidence,
path-lineage summary, observation order/times, filesystem hints, text-fingerprint
method/result, triggering and non-triggering signals, input observation IDs, and
direction rationale. It stores no repeated long source text.

If several rules match the same candidate identity, one row aggregates all rule
results in `signals_json`. `rule_id` is the primary explanation selected by fixed
precedence R1, R2, R3, R4, then R5; the identity intentionally omits rule ID so
the same logical detector revision cannot create duplicate rows for multiple
explanations.

R3/R5 consume only authoritative active PaperVersion membership through the
Paper-membership read port. Gate 2 implements and unit-tests those rules but does
not create Paper membership; before Gate 3 has safely migrated or confirmed any
such membership, the production port returns no inputs for R3/R5. R1, R2, and R4
remain fully operational in Gate 2. This is a dormant deterministic input path,
not an unfinished UI control or inferred association.

Comparison is bounded to at most two direct continuity neighbors, the five
nearest first-observation neighbors inside a Paper, and the five nearest
filename-lineage neighbors. “Nearest” sorts by absolute `first_seen_at`
difference, then candidate snapshot SHA-256 as the total-order tie-breaker; name
neighbors first require equal lineage key and then use the same ordering. A new
snapshot therefore schedules at most 12 distinct pair comparisons after
deduplication. The 10,000-snapshot scale fixture must schedule no more than
120,000 pair comparisons. No all-pairs scan is allowed; pair-count is the
platform-independent performance gate.

Candidate lifecycle is `pending|confirmed|rejected|superseded`. Superseded
requires an explicit `superseded_by_candidate_id`; a detector upgrade alone does
not rewrite or supersede old candidates. Gate 2 is read-only. Gate 3 confirm,
reject, and reconsider are protected commands. Reconsider means only
`rejected→pending`.

Reconsider clears current `decided_at` and `decided_by_command_id`; the rejected
decision remains immutable in AuditChange/DomainEvent history. Supersession sets
only its explicit replacement fields and cannot be produced by reconsideration.

### 10.3 PaperVersion replacement

v0.2 explicitly replaces v0.1 PaperVersion semantics. PaperVersion means one
Paper's formal membership of one immutable SourceSnapshot and contains:

```text
id, paper_id, source_snapshot_id, nullable context_parse_artifact_id,
version_label, normalized_version_label, lifecycle_state,
row_version, created_at, confirmed_by_command_id,
updated_at, updated_by_command_id,
nullable retracted_at, nullable retracted_by_command_id
```

Uniqueness is `(paper_id,source_snapshot_id)` and
`(paper_id,normalized_version_label)`. Label normalization is versioned:
NFC → Unicode outer trim → internal whitespace folding → casefold. Meaningful
digits and punctuation remain. Display label is 1–200 Unicode code points.

`context_parse_artifact_id` is optional and must belong to the snapshot. It is a
protected historical-context pointer; it never follows snapshot preference or
rewrites EvidenceRef. Password-required, failed-parse, and zero-text snapshots
may all be PaperVersions.

PaperVersion lifecycle is `active|retracted`, never soft/physical deletion.
Retraction requires consistent timestamp/command fields and fails with
`VERSION_RETRACTION_DEPENDENCY_CONFLICT` while current, used by an active
Submission, or connected to incompatible active dependencies.

### 10.4 Formal DAG

`version_successor_of` is an asymmetric EntityRelation with:

```text
source = later active PaperVersion
target = earlier active PaperVersion
```

This direction is identical in storage, commands, events, UI arrows, exports,
and cycle checks. Both versions belong to the same Paper. Active edges form a
per-Paper DAG; multiple predecessors and successors are allowed. Retracted edges
are retained but excluded from the active DAG. Branching/merging does not infer
Paper-level `split_from` or `merged_from`.

Candidate confirmation atomically validates candidate version/status, selects
the target Paper, creates or reuses version memberships without overwriting
labels/context pointers, creates the DAG edge, marks the candidate confirmed,
and optionally sets current version only after an explicit choice. Cross-Paper
snapshot reuse is allowed but requires explicit warning and audit.

## 11. Protected CRUD contracts

### 11.1 Shared fields

Paper, Idea, and Submission add only:

```text
row_version INTEGER NOT NULL DEFAULT 1
created_by_command_id UUID NOT NULL
updated_by_command_id UUID NOT NULL
deleted_by_command_id UUID NULL
```

On creation, created/updated command IDs match. Created command is immutable.
Deleted timestamp/command are null or non-null together. Restore clears both,
updates the latest command, increments row version, and preserves delete history.
Technical/cache changes never touch these fields.

### 11.2 Paper

Fields remain `id,title,status,current_version_id,created_at,updated_at,deleted_at`
plus the shared fields. Title is persisted after Unicode outer trim and is 1–500
code points. Status remains `active|paused|revision|submitted|completed|archived`.
Archived is visible business state, not deletion.

`current_version_id` is nullable and must identify an active PaperVersion of the
same Paper. It does not create graph edges. Retraction first switches or clears
the pointer. Active Submission, Idea-Paper relation, current version, EvidenceRef,
or other active formal relation blocks soft deletion with
`DELETE_DEPENDENCY_CONFLICT`; no cascade/detach occurs.

### 11.3 Idea

Fields remain `id,title,content,status,origin_type,created_at,updated_at,deleted_at`
plus shared fields. Title is trimmed and 1–500 code points. Content is validated
nonblank without destructive trim, normalized only from CRLF/CR to LF, and
limited to 1,000,000 Unicode code points. Status remains
`unused|used|parked|archived`. Gate 3 creation fixes `origin_type=manual`; other
v0.1 enum values remain reserved and implement no generators.

Markdown is stored unchanged apart from newline normalization. Rendering disables
raw HTML, script/style/iframe/embed, automatic remote resources, `javascript:`
and `data:` schemes. External links open only after explicit user action. JSONL
keeps Markdown; CSV treats it as text. Active relations or EvidenceRefs block
soft deletion.

### 11.4 Submission

Fields remain `id,paper_id,venue,status,submitted_at,deadline_at,
active_version_id,created_at,updated_at,deleted_at` plus shared fields. Venue is
trimmed and 1–500 code points. `paper_id` is not normally editable. A dedicated
protected correction may reassign only before submission and with no active
version, EvidenceRef, or incompatible relation.

`active_version_id` must be an active version of `paper_id`; it is optional for
preparing/ready and required from submitted onward. `submitted_at` means first
submission: null for preparing/ready and required for every later state,
including withdrawn. It is not overwritten after revision resubmission. Past
times are allowed; an obviously future time is rejected unless within a 5-minute
clock-skew allowance. Deadline may precede creation or submission.

Allowed transitions are:

```text
preparing → ready
ready → preparing | submitted
submitted → editorial_review | external_review | revision | accepted |
            rejected | withdrawn | no_response
editorial_review → external_review | revision | accepted | rejected |
                   withdrawn | no_response
external_review → revision | accepted | rejected | withdrawn | no_response
revision → submitted | editorial_review | external_review | accepted |
           rejected | withdrawn | no_response
no_response → editorial_review | external_review | revision | accepted |
              rejected | withdrawn
rejected → revision
accepted → withdrawn  (requires explicit confirmation)
withdrawn → no ordinary transition
```

Unsubmitted abandonment uses soft deletion, not withdrawn. SubmissionRound is
out of scope; revision/resubmission detail remains in audit history.

## 12. Application commands, audit, undo, and recovery

### 12.1 ApplicationCommand and AuditChange

Every user-visible domain write is one immutable command with ID, type, contract
version, idempotency key, actor, PermissionContext snapshot, lifecycle status,
timestamps, recovery point, optional `undo_of_command_id`, request fingerprint,
result summary, and stable error code. Status is
`pending|running|committed|failed|cancelled`.

One command owns one or more immutable AuditChange rows. Each change stores
entity type/ID, operation, before/after versioned domain snapshots,
`changed_fields`, before/after row versions, and timestamp. Snapshots use closed
domain schemas, never ORM dumps. Large unqueryable all-entity audit JSON is
forbidden.

The semantic command fingerprint is SHA-256 over RFC 8785 normalized request
content after excluding transport ID/timestamps/idempotency key. Same key and
fingerprint returns the prior result. Same key with different fingerprint fails
`COMMAND_IDEMPOTENCY_CONFLICT` without a domain write.

Updates use one atomic statement constrained by expected `row_version`; zero
affected rows means `CONCURRENT_MODIFICATION`.

The coordinator first inserts the privacy-safe command envelope as
`pending/running`, then creates a RecoveryPoint row referencing that command.
After filesystem verification/promotion, one short coordinator transaction
updates the command's `recovery_point_id` and the recovery slot metadata before
the domain transaction may begin. A failed domain transaction rolls back every
domain row, AuditChange, and business DomainEvent; a separate bounded failure
update marks only the existing command `failed`. This resolves the reciprocal
references without deferrable-FK assumptions and never fabricates a successful
business fact.

### 12.2 Compensating undo

Undo is a new protected command, not database replacement. It restores only the
original command's changed domain fields and creates new versions, timestamps,
AuditChanges, and DomainEvents. It never rewinds technical/audit fields.

- Undo create soft-deletes only when no later incompatible dependency exists.
- Undo update permits later changes to disjoint fields and conflicts on the same
  fields or dependent relations.
- Undo delete restores visibility while preserving delete/restore history.
- Undo relation confirmation retracts the formal relation and returns the
  candidate to pending only when safe.
- Batch commands compensate atomically as a whole.
- Undo cannot itself be undone; redo and force override do not exist.

Stable failures include `UNDO_NOT_AVAILABLE`, `UNDO_ALREADY_APPLIED`,
`UNDO_CONFLICT`, `UNDO_DEPENDENCY_CONFLICT`, and
`UNDO_CONSTRAINT_VIOLATION`.

### 12.3 Two-slot pre-command recovery

Protected commands are serialized per workspace:

```text
authorize and validate
→ acquire write/backup lock
→ SQLite Online Backup to same-volume staging
→ fsync and validate
→ rotate current to previous
→ promote staging to current
→ begin domain transaction
→ domain + AuditChange + DomainEvent
→ commit or rollback
→ release lock
```

Recovery validation requires Online Backup completion, closed/flushed file,
directory flush where supported, `PRAGMA integrity_check=ok`, allowed Alembic
revision, required tables, database file hash, manifest match, and same-volume
promotion. Failure means no domain transaction and the exact UI message:

> 无法创建安全恢复点，本次修改未执行。现有数据未发生变化。

The manifest records command ID/type, request fingerprint, schema revision,
database SHA-256, deterministic database snapshot-inventory digest, snapshot
count, and creation time. It verifies snapshot record identity, relative path,
recorded hash, size, existence, and sorted manifest digest; it does not rehash
all snapshot bytes. Directly involved anomalous snapshots may be rehashed.

Recovery generations are monotonically increasing per workspace. The first
verified point promotes staging directly to `current`; `previous` remains empty.
For later generations, the physical protocol is:

```text
verify staging generation N
→ rename existing previous to a same-parent retired temporary name
→ rename current to previous
→ rename staging to current
→ fsync the recovery parent where supported
→ persist slot-state generation/IDs
→ remove retired temporary data
```

Every directory manifest contains recovery-point ID and generation. Startup
reconciliation validates all current/previous/staging/retired candidates. If a
crash occurs before rotation, current remains authoritative; between the two
promotion renames, verified staging becomes current and verified previous remains
fallback; after both renames, current/previous are accepted. A candidate with
wrong ID/hash/generation is never promoted. At least one verified image must
survive every accepted state transition.

RecoveryPoint rows are historical metadata. `recovery_slots` identifies the two
files currently guaranteed to exist. When a slot is overwritten, the older
point becomes `superseded` but its command/manifest/hash history remains; an old
command reference therefore does not promise that its physical image is still
retained. The product promise is the current and previous verified generations.

One application command creates one recovery point and one transaction, not one
per row. Triggering commands are Paper/Idea/Submission create/update/soft-delete,
Submission transition/correction, relation confirm/reject/reconsider/retract,
PaperVersion/current/context changes, undo, and formal-state batch operations.
Queries, navigation, logging, scans, non-persisted calculation, heartbeat,
technical attempt/progress, startup, and seed initialization do not trigger it.

Performance acceptance is less than one second target for a typical 100 MB
database on the reference host defined in §25.4: across five runs, median
creation-and-validation time is `<1.0 s` and the slowest run is `<1.5 s`. At
500 MB, actual time and peak memory are recorded; no UI-heartbeat interval may
exceed 250 ms and Windows must not report the application as not responding.
Backup/integrity work never runs on the Qt thread. Submit controls display
“正在创建安全恢复点” and prevent repeated submission. Future storage optimization
may change mechanics only if it preserves the same pre-command safety semantics.

A restored database contains historical RecoveryPoint rows but Gate 4 does not
copy the internal `recovery/` directory. On first successful activation after a
restore, the coordinator therefore marks every prior point
`physical_state=historical_unavailable_after_restore`, clears `recovery_slots`,
and records one maintenance fact. The next protected command must create and
verify a fresh `current` slot before its domain transaction. No historical
command, audit row, or DomainEvent is rewritten.

## 13. Deterministic execution, authorization, retry, and events

### 13.1 Three execution record classes

1. **ApplicationCommand:** user domain intent, protected where listed, audited,
   event-producing, and potentially undoable.
2. **BackgroundOperation/OperationAttempt:** bounded deterministic I/O or
   computation, optionally correlated to a command, never in the user undo stack.
3. **Raw technical observation:** RawFileEvent and PendingPathCheck, with their
   own state and retention semantics.

The Application layer authorizes and freezes a work plan; a worker performs only
declared I/O/calculation and returns an immutable DTO; the Application layer
validates the DTO; the workspace write coordinator owns Session, transaction,
idempotency, constraints, audit, events, and technical state persistence.
Workers never own Session, QWidget, formal event publication, domain state
transitions, arbitrary paths, or authority to widen a plan.

### 13.2 CapabilityRegistry 1.0

Request capabilities are:

```text
source.observe.request       source.snapshot_import.request
document.parse.request       version_candidate.detect.request
paper.write                  idea.write
submission.write             relation.review
undo.execute                 export.request
backup.request               restore.prepare
maintenance.verify.request
```

Execution roles are closed separately:

```text
source_observer | snapshot_importer | document_parser | candidate_detector |
reconciler | export_worker | backup_worker | restore_worker | verifier
```

A role is bound to one frozen operation plan; it is not a transferable
capability token. `task_executor` and `agent` actors fail `ACTOR_NOT_ENABLED` at
the outer boundary before recovery, staging, or mutation. TaskContract 1.0 and
TaskResult 1.0 remain readable and schema-valid but cannot trigger execution.

PermissionContext 1.0 is an immutable audit snapshot, not a credential. It stores
actor/workspace, capabilities, entity scopes, policy version, decision ID,
grant time, `network_allowed=false`, and structured path-scope summaries:
scope type, normalized-path hash, root ID, access mode, and recursive flag. It
contains no token, OS handle, secret, research content, excerpt, or portable full
path. Historical contexts cannot authorize a new request.

`network_allowed=false` prohibits sockets, HTTP, cloud OCR, remote resources,
and external services. User-approved mapped drives or UNC filesystem paths may
be accessed only through their explicit read/list/copy path scope; this does not
authorize application-layer network protocols.

Long operations revalidate final path identity, emptiness, non-reparse status,
and scope immediately before promotion. Permission denial creates no recovery,
AuditChange, or business event.

### 13.3 RetryPolicyRegistry 1.0

Retry is keyed by stable product error code, never an exception string. Each
entry defines retryability, maximum attempts, backoff, required revalidation,
visible state, and terminal state. Temporary examples are `SOURCE_BUSY`, bounded
`SOURCE_UNSTABLE`, `MONITOR_ROOT_DISCONNECTED`, and safe-boundary
`SQLITE_BUSY`. Permission, validation, unsafe path, password, corruption, hash,
idempotency, user-confirmation, undo, and backup source-integrity failures do not
retry automatically.

SQLite busy retry occurs only before side effects or inside a safely restartable
transaction. Unknown commit outcome is reconciled by command ID, idempotency
key, and database facts. Unknown atomic-promotion outcome verifies staging/final
paths, operation identity, manifest, hashes, and registration before choosing
completed, resume verification, safe cleanup, or manual attention. It never
blindly recopies or overwrites immutable output.

## 14. Gate 4 backup and restore

### 14.1 Package and state

A completed backup is an immutable directory `<label>--<backup-id>.rwsbackup`:

```text
manifest.json
manifest.sha256
workspace.db
sources/...
derived/...
verification.json
```

Identity is UUID; label is metadata. State is
`planned→staging→copying→verifying→completed`, with terminal
`failed|cancelled|incomplete`. Incomplete work is clearly suffixed
`.rwsbackup.incomplete` or kept under staging and cannot be selected for restore.
Completed packages are never overwritten.

Backup and export labels are trimmed NFC text of 1–200 Unicode code points and
never determine identity. Their filename slug uses NFC, replaces separators,
control characters, and Windows-invalid characters with `_`, folds runs of `_`,
strips trailing dots/spaces, protects Windows device names, and truncates to 80
code points. An empty slug becomes `backup` or `export`; the full UUID is always
appended. Existing final names fail rather than overwrite.

SQLite Online Backup is created first. That database is the sole reachability
view. Required snapshots and derived artifacts are then physically copied and
fully rehashed. Hard links are forbidden. Missing, size-mismatched, or
hash-mismatched required content fails `BACKUP_SOURCE_INTEGRITY_FAILED`; no
degraded complete backup exists.

### 14.2 Manifest and verification

`manifest.json` is the normative inventory and uses canonical JSON. It records
package/format/application/schema versions, workspace and backup IDs, null parent
backup ID, actor and creation command, source-workspace path hash, options,
counts, bytes, and deterministically sorted file entries. Entries contain
relative path, role, size, SHA-256, required flag, and optional logical reference.
Roles are `workspace_database|source_snapshot|parse_artifact|
derived_required|derived_rebuildable|backup_metadata`.

Manifest does not include itself. Generation order is content copy/verify,
manifest, manifest hash, final verification, verification report, then atomic
promotion. `verification.json` is a report, never a trust root. Restore always
revalidates independently.

Explicitly classified rebuildable unreferenced caches may be excluded. All
SourceSnapshots and all database-reachable ParseArtifacts, ParseAttempts,
SourceDocuments, ParsedBlocks, EvidenceRefs, PaperVersions, audit/events,
TaskResult/TaskEffect references, legacy rows, and required derived files are
included. Gate 4 uses a destination
free-space requirement of estimated bytes × 1.10, including staging and metadata.

### 14.3 Restore

Restore supports only:

```text
verified .rwsbackup → new strictly empty directory → full copy/revalidation
→ restored-ready marker → pending_data_directory → restart activation
```

It never overwrites, merges, or depends on external source paths. Targets must be
absent or truly empty including hidden/system files, writable, SQLite-capable,
and outside current workspace, package, monitoring roots, and internal paths.
Staging is on the destination volume.

Packages are untrusted. Preflight rejects absolute/traversal paths, duplicate,
casefold-colliding or Unicode-colliding paths, NTFS device names, alternate data
streams, links/reparse points, excessive depth/length, undeclared required files,
and manifest count/size mismatch. Only ordinary listed files/directories copy.
Every hash and database integrity/schema/required-table check reruns.

The v0.2 hard package bounds are at most 1,000,000 manifest entries, 16 TiB
declared total bytes, 64 relative-path segments, and 240 UTF-16 code units per
normalized relative path. These limits are checked before allocation or copy;
destination free-space and filesystem limits are checked independently.

Equal schema restores normally. An older supported schema restores first,
preserves the pre-migration restored database, then applies an approved forward
migration disclosed in preview. A newer unsupported schema fails
`RESTORE_SCHEMA_TOO_NEW`. Bootstrap lightly revalidates and falls back to the
prior active directory if activation fails.

Restore maintenance audit is outside the immutable package and does not rewrite
historical DomainEvents. The restored workspace may record a new system
maintenance fact only after first successful activation.

## 15. Gate 4 research-data export

### 15.1 Package identity and content

A completed export is `<label>--<export-id>.rwsexport` with canonical manifest,
manifest hash, JSONL datasets, and CSV projections. Manifest declares
`package_type=research_workspace_export`, `restorable=false`, and
`contains_source_files=false`. Restore rejects it with
`RESTORE_PACKAGE_TYPE_UNSUPPORTED` based on manifest, not extension.

JSONL is authoritative. Each UTF-8 LF-terminated line is one canonical object
with entity schema version, type, stable ID, and data. Records sort by stable ID.
CSV is a lossy readable projection using UTF-8 BOM, RFC 4180 quoting, fixed
columns, textual UUID/time/enum, lowercase booleans, canonical JSON nested values,
and fixed null/empty encoding. A nullable null is an unquoted empty field; an
empty string is a quoted empty field. The per-dataset CSV schema in manifest
identifies nullable columns and repeats this encoding. Spreadsheet programs may
erase that raw distinction, which is why CSV is not a round-trip contract.

CSV formula protection checks past leading whitespace/control characters for
`=,+,-,@` and prefixes exactly one apostrophe. Every formula-protected textual
column has an adjacent boolean `<column>__formula_escaped` column. That flag is
`true` only when the exporter inserted the prefix, making removal unambiguous
even when user text already began with an apostrophe. The manifest records this
strategy and companion-column contract. JSONL always retains the original text.

### 15.2 Default, opt-ins, and closure

Default export includes visible Paper, Idea, Submission, active formal
relations, associated EvidenceRefs, and source-catalog metadata. The source
catalog includes snapshot hash, original filename, extension/MIME, size, parse
revision identity, and `content_included=false`; it includes no source bytes.

Default excludes deleted objects, historical/retracted/superseded relations,
commands/audit, failed parse attempts, recovery/task/log/cache data, storage
paths, and full external paths. Optional controls are exactly:

1. `include_deleted_and_historical_relations`;
2. `include_full_source_paths`, off by default with explicit privacy confirmation;
3. `include_saved_evidence_excerpts`, off by default with count/character preview.

The first option includes soft-deleted Paper/Idea/Submission rows, retracted or
superseded formal relations, rejected/superseded version candidates, historical
relation/candidate lifecycle fields, and source-catalog entries for referenced
older parse revisions. It still excludes ApplicationCommand, AuditChange,
failed ParseAttempt, and legacy migration records; it is not an audit export.

When paths are included, every selected observation carries path kind, observed
time, and availability. Evidence always uses its original parse/block/locator
and quote hash; the system never re-extracts a quote for export.

Export roots are exactly either `whole_active_workspace` or a nonempty explicit
set of Paper, Idea, and/or Submission IDs. Scope is then a deterministic
controlled reference closure. Preview reports records automatically included to
avoid dangling references. Symmetric relations emit once in canonical endpoint
order. Missing external paths do not fail export; broken database references do.

For `whole_active_workspace`, every in-scope active relation is selected. For an
explicit root set, the relation seed is exactly every supported active relation
(plus opted-in historical relation) having at least one endpoint in the user's
root set. Both endpoints of those seeded relations are included, but relations
incident only to an auto-added endpoint are **not** recursively selected. The
closure then adds EvidenceRefs targeting any included domain object,
PaperVersions directly referenced by included Papers/Submissions/seeded
relations, and source-catalog entries required by those versions/evidence.
Evidence or source-catalog additions never seed further relations. This
deterministic one-relation-hop rule prevents a small selection from expanding to
an entire connected component while leaving no dangling reference in an emitted
relation.

JSONL includes `row_version` for mutable entities; CSV includes it as a clearly
named technical concurrency column. The three command-reference IDs are excluded
from both default and historical research export because their referenced
commands are not exported.

### 15.3 Consistency and state

Long exports use a temporary SQLite Online Backup read copy, which is not a user
backup. State is `planned→snapshotting→writing_jsonl→writing_csv→verifying→
completed`, with `failed|cancelled|incomplete`. Same-volume staging, file hashes,
canonical manifest, verification, and atomic promotion are required. Export is
read-only and creates no Gate 3 recovery point. v0.2 does not import `.rwsexport`.

## 16. Qt ownership, responsiveness, DPI, and interaction

Every long-lived page and every declared dialog has its own `.ui` file in the
locked tree. Python controllers load `.ui`, bind signals, and adapt immutable
view-model DTOs; they do not construct layouts. Worker threads hold no QWidget,
do not mutate Qt models, and never open dialogs. Cancellation is cooperative;
shutdown joins workers without force termination.

Qt object names use lower camel case and semantic suffixes:

- page/dialog roots: `importsPage`, `monitoringPage`, `versionCandidatesPage`,
  `papersPage`, `ideasPage`, `submissionsPage`, `dataTransferPage`, and matching
  `...Dialog` roots;
- controls: `...Label`, `...LineEdit`, `...TextEdit`, `...ComboBox`, `...Button`,
  `...TableView`, `...TreeView`, `...ProgressBar`, `...CheckBox`, `...DateTimeEdit`,
  `...ScrollArea`, and `...ButtonBox`;
- layouts: descriptive `...Layout`; auto-generated names such as `pushButton`,
  `widget_2`, or `verticalLayout_3` are forbidden.

Minimum main-window size remains 1024×720 logical pixels. Qt high-DPI scaling is
enabled before QApplication creation; fixed pixel font sizing and fixed content
heights are forbidden except approved icons. Every long-lived content page owns
a QScrollArea with resizable content. Tables may scroll internally, but the page
must remain usable when tables are empty or narrow. Dialogs have bounded minimum
sizes and become scrollable when content does not fit.

Automated geometry/smoke tests run offscreen. Manual Windows acceptance is
required at 100%, 125%, and 150% scaling, including a mixed-DPI move between two
monitors. At 125/150%, all labels, forms, status badges, progress displays,
tables, and buttons remain visible/reachable with no clipped text or inaccessible
controls. The mixed-DPI item remains REL-GATE-001 until target hardware exists.

No Gate may ship a disabled control advertising a later Gate. Long actions show
progress, cancellability where safe, and stable error text. Import explicitly
discloses local copying and estimated storage. Export discloses path/excerpt
privacy. Restore discloses that current data is not overwritten or migrated.

Import/parse status display uses exactly these user meanings (localized through
the translation layer, never raw enum text): `已导入，可检索文本`,
`已导入，无可提取文本，需要 OCR`, `已导入，解析失败`, and
`已导入，需要密码`. Zero-block content remains eligible for provenance, Paper
association, version candidates, backup, and restore, but not text similarity.

UUID text, `TS` means UTC RFC 3339 `Z` text, `SHA256` means 64 lowercase hex,

all IDs are primary keys, required fields are `NOT NULL`, and timestamps named

## 18. Migration and compatibility protocol

### 18.1 Gate 1 source-identity replacement

Migration 0002 first renames the v0.1 physical `source_documents` table to
`legacy_source_documents_v01`. SQLite foreign keys from the old PaperVersion and
Note tables remain attached to that renamed legacy table. It similarly renames
v0.1 `evidence_refs` to `legacy_evidence_refs_v01` so old evidence remains
intact, then creates the new ParseArtifact-owned `source_documents` and
EvidenceRef tables. Legacy rows receive migration batch/reason/source-revision
metadata without changing their original values.

This permits Gate 1 parsing without pretending old external-path records are
snapshots and without prematurely implementing the Gate 3 version graph. Old
PaperVersion, Note, relation, and observation records remain readable through
their legacy references until 0004.

### 18.2 Gate 3 PaperVersion replacement

Migration 0004 renames the old `paper_versions` to a temporary staging table,
creates the replacement `paper_versions` and the empty
`legacy_paper_versions_v01`, then classifies every staged row exactly once. A
proven row is copied to the replacement table and entered in an immutable
old-ID→new-ID mapping; an unmappable or ambiguous row is copied only to the
legacy table with its reason. After conservation and FK/graph validation, and
only then, the temporary staging table is dropped. Thus a successfully migrated
row is not duplicated in legacy preservation.

Before dropping that staging table, 0004 rebuilds
`legacy_evidence_refs_v01` as the same exact value-preserving schema but without
live FKs to the old PaperVersion/SourceDocument tables. Its old IDs are
historical scalar references, not active graph links. The old
`relation_observations` table is then classified/rebuilt under §17.5. This
ordered FK detachment is tested with `PRAGMA foreign_key_check`; no legacy row
is changed merely to permit the staging-table drop.

0004 also stages and classifies every v0.1 `entity_relations` row before
classifying its observations. Endpoint mapping is exact:

- Paper, Idea, Note, Submission, Conference, and Grant IDs map only when that
  same active/preserved entity row exists;
- PaperVersion IDs require the immutable old→new PaperVersion mapping;
- SourceDocument IDs require a unique succeeded ParseArtifact/SourceDocument
  mapping proven under the snapshot rules;
- EvidenceRef IDs require an explicitly created new EvidenceRef mapping; because
  0002 does not invent such mappings, legacy-only EvidenceRefs normally make the
  relation legacy-only;
- any unsupported entity/relation type, missing endpoint, or ambiguous mapping
  preserves the entire relation in `legacy_dependent_records_v01`.

Asymmetric direction and confidence/confirmation state are preserved. Symmetric
endpoints are canonicalized by `(entity_type, UUID)` and the swap is recorded in
the migration report. If canonicalization produces duplicates, the lowest
relation UUID migrates and each other row is preserved in legacy with
`DUPLICATE_CANONICAL_RELATION`; no row disappears. A migrated historic relation
keeps its ID, actor, created/updated timestamps, confidence, and confirmation
state; it starts with `lifecycle_state=active`, `row_version=1`, nullable
`created_by_command_id`, and null supersession/retraction fields. The adoption
command is not falsely attributed as its creator.

An immutable old-relation→new-relation mapping then controls observation
classification: an observation migrates only when its relation mapping and any
EvidenceRef mapping both exist. Required conservation is:

```text
old EntityRelation count = migrated relation count + legacy relation count
old RelationObservation count = migrated observation count + legacy observation count
```

Additional stable reasons are `UNMAPPABLE_RELATION_ENDPOINT`,
`INVALID_LEGACY_RELATION_TYPE`, and `DUPLICATE_CANONICAL_RELATION`.

Proof requires an existing registered SourceSnapshot whose verified SHA-256
matches the old record or an approved explicit one-to-one import mapping. Name,
path, ID, size, or timestamp alone is insufficient. Ambiguity stays legacy.

After safely mapped version rows exist, a valid old `parent_version_id` becomes
`source=child/newer, relation=version_successor_of, target=parent/earlier` after
same-Paper, missing-reference, self-link, duplicate, and cycle validation. An
invalid edge is reported without discarding an otherwise valid membership.

One safely mapped `is_current=true` sets Paper.current_version. Zero leaves null.
Multiple or unmappable current flags leave null and report ambiguity. No time or
label heuristic chooses current.

Existing Paper/Idea/Submission rows receive command-reference fields through one
clearly identified system maintenance command
`system.migration_adopt_v01` per migration batch. The ordered protocol is:

```text
create migration batch and verified pre-migration Online Backup
→ create ApplicationCommand/AuditChange-era tables
→ insert committed system.migration_adopt_v01 command linked to migration batch
→ rebuild/adopt rows using that command ID
→ validate counts, constraints, and startability
→ mark migration batch committed
```

This maintenance command has nullable `recovery_point_id`: the migration
batch's verified pre-migration backup hash is its protection proof. It is
excluded from user undo and emits maintenance audit only—not fabricated
Paper/Idea/Submission business events.

### 18.3 Conservation and rollback

Every structural migration performs SQLite Online Backup before DDL, runs
transactionally where SQLite supports it, and uses explicitly tested staged
table rebuilds otherwise. Failure leaves the prior database startable and keeps
the verified pre-migration image. Source snapshots are untouched.

Required reconciliation includes:

```text
old PaperVersion count = migrated PaperVersion count + legacy PaperVersion count
```

It also accounts for old source documents, evidence, dependent links, parent
edges, current flags, labels, timestamps, relation observations, mapping
conflicts, and legacy reason
counts. Labels are never silently renamed; snapshots are never invented; demo
or unknown rows are never deleted merely because their IDs are recognized.

## 20. Dependency and commercial-license gate

Gate 1 parser choices are `python-docx` (MIT), `pypdf` (BSD-3-Clause), and
`python-pptx` (MIT). Gate 2 adds `watchdog` (Apache-2.0). Each is subject to the
actual resolved version and transitive dependency inventory in the Gate that
introduces it. PyMuPDF is prohibited for
v0.2 because its AGPL/commercial dual-license path is outside the approved
baseline. Docling, MarkItDown, OCR engines, and external converters are absent.

The lockfile, installed distributions, license texts, notices, and SPDX inventory
are generated and tested before adapter production code is accepted. New runtime
dependencies must be permissive or already approved under the foundation's Qt
LGPL compliance path. AGPL, SSPL, source-available/noncommercial, GPL-only,
unknown, missing, or conflicting licenses are a hard stop. Tests verify both
direct and transitive packages and package the required notices. A dependency
version change reruns the same gate; a permissive project name is not sufficient
evidence for a different resolved artifact.

## 21. Privacy and source safety

- Original sources and committed snapshots are never modified by workflows.
- Full paths may exist only in protected local provenance/operation data and
  explicit path-enabled export; they do not enter ordinary logs, events,
  portable PermissionContext, or default export.
- Logs use operation IDs, entity IDs, path hashes, byte counts, durations, and
  stable error codes. They exclude titles/content when not required, excerpts,
  parser text, PDF passwords, raw exception payloads containing paths, secrets,
  and connector credentials.
- Backups contain the local database and therefore user data; UI treats their
  destination as sensitive. Exports explicitly disclose plaintext/path/excerpt
  choices.
- Worker diagnostics are schema-filtered before persistence. A failed permission
  record stores command/type/actor/policy/time/fingerprint/error only.
- No built-in v0.2 workflow opens outbound sockets or remote resources.
